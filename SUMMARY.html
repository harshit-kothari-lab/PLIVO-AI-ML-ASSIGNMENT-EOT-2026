Plivo · Voice-AI take-home

Knowing when the caller is done talking.

An end-of-turn detector built from scratch — no pretrained models — that decides, at every silence, whether the speaker has finished, using only the audio it has already heard.

A live voice agent that waits a fixed 1.6 seconds of silence before replying feels slow. The job here is to cut that wait without talking over people: for each pause emit p_eot, the probability the turn is over, then let the agent respond as soon as it is confident. The one hard rule is causality — a feature may read audio[0 : pause_start] and not a sample more, because in production the future does not exist yet.

The whole problem in one strip: everything left of the red line is fair game; everything right of it is cheating.

01 / RESULT

What it scores

The metric (from the provided score.py) simulates the live agent: it sweeps a confidence threshold and a silence-delay budget, and reports the mean response delay in milliseconds at a ≤5% false-cutoff rate — how fast the agent replies while interrupting no more than 5 turns in 100. Lower is better; the silence-only reference is ~1600 ms.

Why Hindi's latency sticks at 850 ms while its AUC jumps: on this particular set Hindi holds are short, so "wait 850 ms and fire on everything" is already near-optimal and hard to beat on latency alone. The detector's much cleaner ranking of pauses doesn't lower the dev latency here, but on the hidden, mostly-Hindi set — different turns, different hold lengths — a 0.71-AUC model beats a coin-flip baseline in a way the 850 ms floor hides.

02 / THE SIGNAL

What actually separates a pause from an ending

Before writing a classifier I measured which acoustic cues move. The dominant one is pitch: heading into a genuine end-of-turn a speaker's F0 falls and settles a few semitones below their own turn baseline; heading into a mid-thought hold it stays level or rises. That difference is large and clean in Hindi and weak — even slightly reversed — in English, which is the whole reason English is the harder language and why its baseline is so high.

Around that spine sit energy release and decay, final-syllable lengthening, trailing voicing and creak, speaking-rate, and coarse position context. All 35 features are summaries of the last ~1.5 seconds before the pause.

03 / METHOD

How it's built

Causal by construction

Per-frame contours (energy, pitch, spectrum) are computed once per file with frame-local operators only — librosa's yin, rms, and stft, never pyin's Viterbi pass, which would let a frame's estimate borrow from future audio. For each pause I then keep only the frames whose analysis window ends before pause_start, and the per-turn pitch baseline is taken over that same prefix. The clean pitch on the trailing 1.5 s is run on a truncated pre-pause slice. pause_end is never read. A grader can confirm all of this by reading one file, predict.py.

The model

A small ensemble — two gradient-boosted trees plus a scaled logistic regression, blended as (2·mean(GBM) + LR)/3. Trees capture the language-specific interactions; the linear model steadies the ranking. Every choice was selected by grouped cross-validation on the real metric, not on accuracy.

A quiet but decisive insight: the scorer sweeps the threshold, so the score depends only on the ranking of pauses — not on calibrated probability values. That single observation killed a whole branch of tempting work (probability calibration, which actively hurt) and reframed the goal as pure ranking quality.
04 / DEAD ENDS

What I tried that didn't work

  • Up-weighting long holds (the costly false positives) to force them down the ranking — lowered AUC instead of helping.
  • Per-language models — starved each of data; the combined model, with turn-relative features that normalise away the language baseline, generalised better.
  • Probability calibration (isotonic/sigmoid) — overfit on 100 turns and, because the metric is rank-based, could only hurt.
  • Dropping pause_start as a robustness check left AUC unchanged (0.68 / 0.71) — proof the model leans on real prosody, not a positional shortcut that might not survive the hidden set.
05 / HONESTY

The number I did not report

Train the final model on both folders and score it on those same folders and you get 145 ms at AUC 0.997. It is meaningless — the model has seen every answer. This is exactly the illusion the brief warns about.

Every headline figure in this report is grouped cross-validation by turn: each turn is scored only by models that never saw it. The delivered predictions_english.csv and predictions_hindi.csv are those out-of-fold predictions, so their scores (1200 ms / 850 ms) are trustworthy. predict.py — which carries the all-data ensemble inline — is what runs on the unseen test set.
06 / DIVISION OF LABOUR

Human vs. coding agent

The assignment is graded on what a person adds beyond what a coding assistant produces on its own. The split was deliberate.

The human judgement

  • Measuring the data first and discovering the pitch-finality cue — and that it's strong in Hindi, weak in English.
  • Realising the scorer is rank-determined, so calibration is pointless and long holds are the enemy.
  • Designing the causality guard and defending it against pyin's hidden future-leak.
  • Insisting on grouped-CV and refusing the seductive 145 ms in-sample score.
  • Reading why the Hindi floor exists rather than chasing it.

The coding agent's leverage

  • Boilerplate: argparse, CSV I/O, the pickle round-trip.
  • Correct librosa calls and framing arithmetic.
  • The cross-validation harness and the matplotlib charts.
  • Refactoring features into one clean, auditable module.

Put differently: the agent wrote the code quickly; the edge came from choosing what to compute, how to evaluate it honestly, and which results to disbelieve.
